## Part 1: Epic 2 Review

### What Went Well

**1. Scaffolding-Then-Integrate Strategy Was Validated**

The approach of building custom scaffolding (Stories 2.3-2.4) and then replacing it with the official `@crosstown/client` library (Story 2.5) proved highly effective. The scaffolding gave the team deep understanding of the publish pipeline internals, which made the integration story smoother and more reliable.

Charlie (Senior Dev): "Building event-signing.ts and crosstown-connector.ts from scratch forced us to understand the ILP packet flow intimately. When @crosstown/client arrived, we knew exactly what it should do and could validate it properly."

Elena (Junior Dev): "The scaffolding tests became our integration contract. We knew the adapter was correct because it produced identical behavior to the scaffolding it replaced."

**2. Scope Management Was Proactive and Honest**

Stories 2.6 (Identity Propagation) and 2.7 (ILP Fee Collection) were moved to Epic 3 when the team recognized that BLS-side implementation belongs in the Crosstown repository. This was documented explicitly in sprint-status.yaml with clear rationale rather than being silently dropped.

Alice (Product Owner): "Moving Stories 2.6 and 2.7 to Epic 3 was the right call. We specced them in Story 2.4 so nothing was lost -- the work is documented and ready. We just acknowledged where the implementation boundary really is."

**3. Architectural Decision-Making Under Uncertainty**

Story 2.4 surfaced three major blockers (BLOCKER-1 through BLOCKER-3) that required architectural decisions about BitCraft reducer modification, wallet balance scope, and Crosstown implementation specs. The team resolved all three decisively:
- **BLOCKER-1:** Accepted modifying BitCraft reducers to add identity parameter (superseding "run unmodified" principle)
- **BLOCKER-2:** Removed wallet balance checks from BLS scope (EVM onchain wallets are out of scope)
- **BLOCKER-3:** Created detailed Crosstown implementation specs for the external team

Jonathan (Project Lead): "The BLOCKER-1 decision was significant. Accepting that we need to modify BitCraft reducers was a pragmatic choice that unblocked the entire identity propagation architecture. Purity of 'run unmodified' was less valuable than working identity propagation."

**4. Comprehensive Security Posture Maintained**

Every story passed OWASP Top 10 review (AGREEMENT-2 continued from Epic 1). Five semgrep security issues were found and fixed during the epic. The team maintained the security-first code review protocol established in Epic 1.

Dana (QA Engineer): "The security scanning caught real issues, not just noise. The semgrep findings were legitimate vulnerabilities that could have been exploited."

**5. Test-First Approach Continued Strong**

The TDD approach (AGREEMENT-1) remained effective with 399 story-specific tests written across the epic. Test traceability achieved 100% coverage with all 36 acceptance criteria mapped to specific test cases.

Dana (QA Engineer): "The traceability gate passing at 36/36 ACs with 100% P0/P1 coverage gives us high confidence. Every requirement has a corresponding test."

### What Didn't Go Well (Challenges)

**1. External Dependency Risk: @crosstown/client**

Story 2.5 depended on `@crosstown/client@^0.4.2` being published on npm. This external dependency introduced schedule risk -- if the package had been delayed, Story 2.5 would have been blocked. The team mitigated this by building scaffolding first (Stories 2.3-2.4) but the dependency risk remained real.

Charlie (Senior Dev): "We got lucky that @crosstown/client was available when we needed it. If it had been delayed, we'd have shipped the scaffolding as the production path and integrated later. The scaffolding-first approach was insurance."

**2. @crosstown/client Workspace Stubs**

The `@crosstown/client` and `@crosstown/relay` packages required workspace stubs for the build system. This is a fragile setup -- if the real packages diverge from the stub interfaces, builds will break silently until integration tests catch it.

Elena (Junior Dev): "The workspace stubs worry me. We're assuming the real API matches our stubs, but we don't have a contract test to verify it."

**3. getEvmAddress() Placeholder**

The `getEvmAddress()` method in the CrosstownAdapter returns a placeholder value. This will need to be properly implemented when EVM address derivation is required for payment channels.

Charlie (Senior Dev): "I documented it as a known limitation, but it's the kind of thing that could bite us in Epic 3 if we forget."

**4. Deferred Integration and Performance Testing**

NFR compliance had 2 CONCERNS -- Stories 2.3 and 2.5 deferred integration testing and performance validation to when the full Docker stack with BLS handler is running. This means we have untested paths in the publish pipeline.

Dana (QA Engineer): "103 skipped tests is a lot. They're skipped because they need Docker with a real BLS handler, which we don't have yet. Once Epic 3 delivers the BLS handler, we need to un-skip and validate."

**5. DEBT-4: BitCraft Reducer Modification**

The decision to modify BitCraft reducers (BLOCKER-1 from Story 2.4) creates a new technical debt item. The reducers need to be modified to accept `identity: String` as the first parameter. This work hasn't been done yet and is a prerequisite for end-to-end identity propagation.

Alice (Product Owner): "We accepted this as the right architectural decision, but the actual reducer modifications haven't been implemented yet. That's a critical dependency for Epic 3."

**6. Observability Deferred**

No observability or monitoring was added during Epic 2. ILP packet throughput, fee revenue tracking, and BLS latency monitoring are all deferred to Epic 8. During development, debugging relied on console logs and test output.

Charlie (Senior Dev): "We're building a payment pipeline without any observability. For dev and testing that's fine, but it needs to be addressed before any production use."

### Key Insights

**1. Scaffolding Is a Valid Development Strategy**

Building custom implementations before integrating production libraries provides deep understanding of the problem domain, generates comprehensive test suites, and creates fallback options if the external dependency doesn't materialize. The scaffolding-then-integrate pattern should be used whenever external dependencies have uncertain timelines.

**2. Scope Corrections Are Features, Not Failures**

Moving Stories 2.6 and 2.7 to Epic 3 was a process success. The team recognized that BLS-side implementation is in a different codebase and adjusted scope accordingly, documenting the rationale. This is better than artificially keeping stories in the wrong epic.

**3. Architectural Decisions Should Be Made Explicitly, Not Deferred**

The three blockers in Story 2.4 could have stalled the epic if the team had deferred the decisions. Instead, the decisions were made explicitly (modify reducers, remove wallet checks, create Crosstown specs), documented, and the epic continued. Decisiveness beats perfection.

**4. External Dependencies Need Contract Tests**

The `@crosstown/client` workspace stubs highlight a gap: we don't have contract tests that verify our assumptions about external package APIs. If the real package evolves differently from our stubs, we'll discover it late.

**5. Test Skipping Must Be Tracked and Resolved**

103 skipped tests represent deferred validation. These need a clear owner and timeline for resolution. Skipping tests is acceptable temporarily but becomes technical debt if not addressed.
