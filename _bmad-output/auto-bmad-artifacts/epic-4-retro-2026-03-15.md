### What Didn't Go Well (Challenges)

**1. Missing Story Report Artifact (Story 4.5)**

The `story-4-5-report.md` file was not generated with the standard filename convention (it used `story-4.5-report.md` with a dot instead of a hyphen). While the data was reconstructable from other artifacts, this inconsistency in naming conventions caused confusion during the aggregation pipeline. The pipeline should enforce consistent naming.

Dana (QA Engineer): "Inconsistent artifact naming is a process smell. It didn't cause any data loss, but it added unnecessary friction to the aggregation step. We should enforce the naming convention in the pipeline."

**2. gray-matter npm Audit Concern**

Story 4.1 introduced the `gray-matter` library (^4.0.3) as a YAML frontmatter parser. This dependency was flagged as a potential audit concern. While no known vulnerabilities exist at the time of implementation, adding a new npm dependency to the client library increases the supply chain surface area. The library is well-maintained (2M+ weekly downloads) but the concern was documented without resolution.

Charlie (Senior Dev): "gray-matter is the standard library for this use case, and it has broad adoption. But every dependency is attack surface. We should verify it passes npm audit as part of CI."

**3. Budget Guard Not Yet Wired into Publish Pipeline**

Story 4.4 created the `BudgetPublishGuard` class for pre-publish enforcement, but it is not yet composed into the actual `client.publish()` flow. The guard is ready for integration but requires wiring in a future epic (likely Epic 6 MCP integration). This is documented as intentional but represents a gap between the component and its runtime usage.

Alice (Product Owner): "The budget guard is tested and works in isolation, but no agent actually uses it yet. Until it is wired into the publish pipeline, it is a library function waiting for a caller. Epic 6 will close this gap."

**4. Event Interpreter Has Limited Table Coverage**

Story 4.5's EventInterpreter ships with interpreters for only 4 tables (player_state, entity_position, inventory, resource_spawn). The BitCraft game module has many more tables. Epic 5 (BitCraft Game Analysis) will likely require additional table-specific interpreters, which means the interpreter system will need extension.

Elena (Junior Dev): "Four tables is the minimum viable set. We have the graceful fallback for unmapped tables (AC4), but the narratives for unknown tables are generic and not very useful. Epic 5 will tell us which tables matter most."

**5. Silent Error Swallowing in Decision Logger**

Story 4.6's `DecisionLogger` silently swallows all file I/O errors. This is intentional (an agent must not crash due to logging failures) but means logging failures are invisible. There is no callback, event, or metric for logging errors. For long-running experiments, silent logging failures could result in data loss without notification.

Charlie (Senior Dev): "Silent swallowing is the right default for MVP -- you never want your logging system to crash your agent. But we should add an optional error callback or event emission in a future iteration so researchers can detect logging failures."

**6. Double File Read in Versioned Config Loader**

Story 4.7's `readSkillContents()` reads all skill files a second time (after `loadAgentConfig` already parsed them) to compute content hashes. This is an accepted inefficiency for MVP but represents wasted I/O for large skill directories. A future optimization could cache file contents during the initial parse.
