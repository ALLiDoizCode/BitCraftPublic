**2. Static Data Table Gap Still Unresolved (DEBT-2)**

Story 5.2 found that only 34 of 108 static data tables are loaded in the SDK, with placeholder names that don't match actual server table names. This gap affected Stories 5.6 (resource gathering) and 5.7 (crafting), where tests had to use discovery-driven approaches and graceful degradation instead of direct static data lookups.

Charlie (Senior Dev): "DEBT-2 is becoming a recurring theme. It was flagged in Story 1.5, documented again in Stories 5.2, 5.6, and 5.7. The static data gap doesn't block validation tests (discovery-driven testing works), but it will block the MCP server if agents need to look up item names or recipe requirements."

**3. Docker-Dependent Tests Cannot Run Without Docker**

All 129 integration tests from Stories 5.4-5.8 are Docker-dependent and skip when Docker is unavailable. The total skipped test count grew from 242 (Epic 4 baseline) to 378. While graceful skipping is the correct behavior (AGREEMENT-5), it means that CI without Docker validates only the documentation verification tests (Stories 5.1-5.3), not the integration tests.

Dana (QA Engineer): "We went from 242 skipped to 378 skipped -- that is 136 new Docker-dependent tests. In CI without Docker, these are invisible. We need Docker in CI for Epic 5's tests to provide any regression protection."

**4. Fresh Game World State Uncertainty (R5-030, R5-031)**

Stories 5.6 and 5.7 documented that a fresh game world may lack player-constructed buildings, recipes, or resources necessary for full crafting validation. Tests handle this gracefully with discovery-driven fallbacks and diagnostic output, but the degraded test execution provides less validation confidence.

Elena (Junior Dev): "The crafting tests in Story 5.7 are the most fragile because they depend on game world state that may not exist in a fresh environment. The discovery-driven approach is correct, but we should consider a world seeding solution for CI."

**5. Docker Pause/Unpause Unknown in CI (R5-040, R5-042)**

Story 5.8's connection loss tests use `docker pause` and `docker unpause` to simulate SpacetimeDB disconnection. Whether this works in CI environments (GitHub Actions) is unknown. Tests skip gracefully when Docker control commands fail, but the connection loss validation may be effectively disabled in CI.

**6. Test Count Reporting Inconsistency**

The final passing test count is 1,760 (from the pipeline data), but Story 5.8's report says 1,752 at its regression step. The delta of 8 tests is due to the final project-wide count including tests from the seeding utilities created as infrastructure. ### Key Insights

**1. Research Epics Require a Different Velocity Model**

Epic 5's research stories (5.1-5.3) took longer than typical implementation stories because they involved deep analysis of Rust source code, cross-referencing multiple files, and producing structured documentation with verification tests. However, the validation stories (5.4-5.8) were faster because the research phase had already mapped the territory. The pattern of "research first, validate second" should be expected to have higher per-story cost in the research phase and lower cost in the validation phase.

**2. Document Verification Tests Are a Powerful Pattern**

Stories 5.1-5.3 introduced a novel test pattern: tests that verify the structure and completeness of documentation, not code behavior. These 324 verification tests ensure that the BitCraft Game Reference contains the required sections, counts, relationships, and cross-references. If the document is edited in the future, the tests catch structural regressions. This pattern should be adopted for any reference documentation produced in future epics.

**3. BSATN Serialization Is the Integration Test Bottleneck**

Stories 5.4-5.8 revealed that the most time-consuming aspect of writing integration tests is crafting correct BSATN (Binary SpacetimeDB Application Transmission Notation) payloads for reducer calls. The `test-client.ts` fixture accumulated serialization functions for 10+ reducers (synchronize_time, sign_in, sign_out, player_queue_join, player_move, extract_start, extract, craft_start, craft_continue, craft_cancel, chat_post_message). Each reducer requires byte-level encoding knowledge. A code generation approach for BSATN serialization would significantly accelerate future integration test development.

**4. Discovery-Driven Testing Handles World State Uncertainty**

The pattern of "discover what exists, then test against it" (used in Stories 5.6 and 5.7) proved effective for handling the unknown game world state. Rather than assuming specific resources, buildings, or recipes exist, tests query the game world first, find suitable test targets, and then exercise the game loop. This pattern should be documented as the recommended approach for all future game system validation epics.

**5. The Pipeline Has Matured to Near-Autonomous Operation**

Across 8 stories and 176 pipeline steps, there were zero pipeline failures requiring manual intervention. The pipeline consistently produced:

- 128 code review issues caught and resolved
- 17 false-positive semgrep findings handled
- 100% traceability at every story
- No cross-story regressions

The pipeline operated as a reliable quality machine for the fifth consecutive epic.

**6. The Epic 5 Pattern Is the Template for Epics 9-13**

The 2026-03-16 restructure explicitly references the Epic 5 pattern (static data -> subscriptions -> reducer validation -> fixtures -> error scenarios) as the template for SDK validation Epics 9-13. The fixture framework, the discovery-driven testing approach, and the document verification test pattern are all transferable. This means Epic 5's investment in infrastructure will compound across 5 future epics.
